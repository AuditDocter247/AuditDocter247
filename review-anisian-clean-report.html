Anisian (ANI) — Verification of Claimed
Security Properties

Anisian.sol Live Contract — Base Mainnet Independent Review, No Client Engagement
No Issues Found
This is a self-directed review of a real, deployed contract, undertaken as independent practice — not a paid engagement. The project's public documentation makes four specific, falsifiable claims about its security properties. Each was individually verified against the actual source code. All four claims were found to be accurate.
Context & Purpose

Anisian (ANI) is a fixed-supply ERC-20 token deployed on Base, publicly described by its team as "fixed-supply, immutable ERC-20 on Base with deflationary halving burn schedule. No admin, no mint, no pause, no upgrade."

Bold, specific security claims like this are exactly the kind of statement worth independently verifying — a claim is only as good as the code behind it. This review sets out to check each stated property directly against the deployed contract's source, rather than take the description at face value.

Claims Verified
  • "No mint" — Confirmed. _mint() is called exactly once, inside the constructor, minting the full 100,000,000 ANI fixed supply. No other function in the contract calls _mint, and no mechanism exists to increase supply post-deployment.
  • "No admin" — Confirmed. The contract does not inherit Ownable or implement any equivalent privileged-caller pattern. The one setup function, initialize(), is restricted to the original deployer, runs exactly once, and permanently locks via an AlreadyInitialized guard — after which no address, including the original deployer, retains any privileged capability.
  • "No pause" — Confirmed. No pause mechanism, modifier, or state flag exists anywhere in the contract that could halt transfers.
  • "No upgrade" — Confirmed. The contract is a standalone, non-proxy deployment with no delegatecall pattern, no implementation-address storage slot, and no upgrade function of any kind.
Initializer Safety — Positive Pattern

Worth specifically noting: initialize() follows the correct pattern for this bug class — unlike a vulnerable proxy initializer (see Finding #003 in this portfolio, where a race condition allowed anyone to claim ownership), Anisian's initializer is protected because _initializer is set inside the constructor, atomically at deployment. There is no window during which an uninitialised, publicly-callable contract exists on-chain.

Anisian.sol — Correctly Guarded Initializer
constructor() ERC20("Anisian", "ANI") {
    _initializer = msg.sender; // ✅ set atomically at deployment — no front-running window
    _mint(msg.sender, INITIAL_SUPPLY);
}

function initialize(address vault, address pool, address ownerWallet) external {
    if (msg.sender != _initializer) revert NotInitializer(); // ✅ caller-restricted
    if (_initialized) revert AlreadyInitialized();      // ✅ one-shot, permanently locked
    _initialized = true;

}
Observation — Not a Finding

One design limitation was identified during review and is documented here for completeness, though it does not constitute a valid finding:

  • The launch-phase buy limits (MAX_BUY_AMOUNT, MAX_WALLET_AMOUNT) are enforced per-address. An actor controlling multiple wallets could distribute purchases across them to exceed the intended per-entity accumulation limit during the 90-day protection window.
Why This Is Not Reported as a Finding Per-wallet rate limiting is a well-understood, universal constraint of any address-based system — no on-chain mechanism can determine that two distinct addresses are controlled by the same individual. The project's own documentation does not claim sybil-resistance; it describes a buy limit, and the buy limit functions exactly as implemented. The applicable audit test — did the contract fail to enforce a guarantee it explicitly made? — is not met here. This is included purely as a transparency note, not a criticism.
Conclusion

Based on this independent review, Anisian's publicly stated security properties are accurate as implemented. The contract demonstrates disciplined engineering: a correctly atomic initializer, a genuinely fixed supply, and the deliberate absence of privileged post-deployment control — properties that are often claimed but rarely fully upheld under direct code inspection.

This review is independent and unpaid, conducted for portfolio and practice purposes. It does not constitute a formal, comprehensive audit and should not be relied upon as one.
